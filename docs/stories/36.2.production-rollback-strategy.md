# Story 36.2 — Production Rollback Strategy (G7)

**Priority:** Medium
**Effort:** Medium
**Type:** Enhancement / Reliability
**Status:** Backlog

---

## Context

Story 36.1 fixed rollback for staging (G6) but production (G7) has **no rollback mechanism**. If the production health check fails after deploy, the workflow exits with error and the broken deploy stays live. This is the last gap in the CI/CD pipeline reliability.

Production rollback is higher risk than staging — it affects real users and revenue. The strategy must be more careful than a simple `git checkout HEAD~1`.

---

## Problem

**Current behavior:** If `deploy-production` health check fails (30 retries over 5 min), the workflow fails but:
- API stays on the broken version (Railway)
- Workers stay on the broken version (Railway)
- Console stays on the broken version (Vercel)
- No alert is sent
- No automatic recovery happens

**Impact:** Downtime until a human notices, investigates, and manually redeploys.

---

## Acceptance Criteria

### AC 36.2.1 — Railway rollback on health check failure
- **Given** the production health check fails after G7 deploy
- **When** the rollback step triggers
- **Then** API and all 3 workers are redeployed from the previous commit using `RAILWAY_TOKEN_PROD` (never the staging token)
- **And** the checkout has `fetch-depth: 2` to ensure `HEAD~1` exists

### AC 36.2.2 — Vercel Console rollback on health check failure
- **Given** the production health check fails after G7 deploy
- **When** the rollback step triggers
- **Then** the previous Vercel production deployment is promoted via `vercel promote`
- **And** a warning is logged if no previous deployment is found

### AC 36.2.3 — Production smoke tests before declaring success
- **Given** production deploy completes and health check passes
- **When** post-deploy validation runs
- **Then** all 5 checks from G6 are replicated against production:
  1. `/health` returns 200 (API + DB connectivity)
  2. `/api/notifications` returns 401/403 (auth guard)
  3. `/api/tenants` returns 401/403 (auth guard on second endpoint)
  4. `PRODUCTION_CONSOLE_URL` returns 2xx/3xx (Console reachable)
  5. `/api/nonexistent-route` returns 404 (error handling, not 500)
- **And** `PRODUCTION_CONSOLE_URL` must be configured as a GitHub Actions variable

### AC 36.2.4 — Alert on rollback
- **Given** a production rollback is triggered
- **When** the rollback completes (success or failure)
- **Then** a notification is sent (GitHub Actions summary + optional Slack/Discord webhook)
- **And** the notification includes: commit SHA, failure reason, rollback target SHA, rollback result

### AC 36.2.5 — Rollback verification
- **Given** a production rollback was executed
- **When** the rollback deploy finishes
- **Then** the health check runs again against the rolled-back version
- **And** if the rollback also fails, the workflow exits with a critical alert

## Technical Notes

### Rollback approach
Reuse the same pattern proven in G6 (staging):
```yaml
- name: Rollback Railway on Failure
  if: failure()
  run: |
    PREV_SHA=$(git rev-parse HEAD~1)
    git checkout "$PREV_SHA" -- .
    railway up --service api --environment production
    # ... workers
```

### Considerations
- **Separate tokens:** Production uses `RAILWAY_TOKEN_PROD`, not `RAILWAY_TOKEN`
- **Vercel production:** Use `vercel promote` (same as staging rollback)
- **Database migrations:** If the deploy included a migration, rollback may leave the DB in a forward state with old code. This story does NOT cover migration rollback — that requires a separate strategy (backward-compatible migrations)
- **Canary/blue-green:** Out of scope for this story. This is a simple "revert to previous" rollback. Progressive delivery is a separate initiative

### Alerting
Default to **GitHub Actions Job Summary** (free, built-in, no secrets needed). Slack/Discord/PagerDuty integration is a follow-up.

### Merge strategy assumption
This rollback uses `HEAD~1`. For this to work correctly, the team **must use squash-merge** for PRs into main. With regular merge commits, `HEAD~1` could point to the last commit on the feature branch instead of the previous release. If the team switches to regular merges, the rollback target logic must be updated.

### Known limitations
- **Worker health is not verified** — Workers are BullMQ processors with no HTTP endpoint. A broken worker deploy won't be detected by the health check. Worker health monitoring (queue depth, job failure rate) is a separate initiative.
- **Partial rollback** — If Railway rollback succeeds but Vercel rollback fails (or vice versa), the system could be in an inconsistent state (API v(N-1) + Console v(N)). Both rollback steps run independently (`if: failure()`). A partial failure will be visible in the workflow logs and the alert (AC 36.2.4).

---

## Out of Scope
- Database migration rollback (backward-compatible migrations are a separate practice)
- Environment variable rollback (if deploy adds new env vars in Railway/Vercel, code rollback without env rollback could cause issues)
- Canary deployments / blue-green strategy
- Automatic rollback based on error rate (Sentry integration)
- Multi-commit rollback (only rolls back 1 commit)
- Worker health monitoring (BullMQ queue depth / job failure rate)
- Slack/Discord/PagerDuty alerting (follow-up to GitHub Actions Job Summary)

## Follow-up suggestions
- Add post-rollback health check to G6 staging (currently only planned for G7)
- Worker health endpoint or queue monitoring for deploy verification
- Progressive delivery (canary/blue-green) for zero-downtime deploys
